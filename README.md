Final Reflection
Difficulties I faced and how I overcame them
The biggest challenge was the sheer number of errors at the start. When I saw 64 errors I wanted to delete everything and start over. Instead I learned to read the error list carefully and look for patterns. Most errors were caused by the same 1 or 2 root problems.
Another difficulty was that the code I was building referenced things that didn't exist yet in my other classes (like BetHistory in Bettor, or RacerNo in Bet). I learned to always check what properties and methods actually exist in a class before trying to use them from somewhere else.
The Git authentication issue at the end was unexpected and stressful, but it reminded me that technical problems always have a solution. You just have to read the message carefully and troubleshoot step by step. But unfortunately that wasn’t even enough. 
How my intentions shaped my approach
Setting an intention at the start of each session actually helped more than I expected. When I intended to be "curious" instead of frustrated, I approached errors differently — like puzzles to solve rather than proof that I wasn't good enough. When I intended to be "patient," I slowed down and read error messages more carefully instead of randomly changing things hoping it would work.
Intentions are useful for difficult challenges because they give you a mental frame before you even start. If you decide in advance how you want to show up, you're less likely to spiral when things get hard.
How resilience changed in me
At the start of this assignment I would see an error and immediately feel defeated. By the end I was reading errors almost calmly, treating them as information. I still felt frustrated sometimes, but I didn't stay there as long. I got better at saying "okay, what is this actually telling me?" instead of "why is nothing working."
Resilience for me isn't about not feeling frustrated. it's about not letting frustration be the last word. This assignment gave me a lot of practice with that.
